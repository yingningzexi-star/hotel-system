## 第一部分：答辩逐页发言模板

> [!TIP]
> **汇报人核心姿态**：沉稳、专业、逻辑清晰。汇报时多强调**“业务闭环”**、**“数据一致性”**与**“团队规范协作”**。

### Slide 1: 封面页 (开场白)
*   **建议发言**：
    > “各位老师好！我们小组本次汇报的课题是《Luxe Stay 酒店在线预订平台》。本项目基于 Spring Boot 3 与 SQL Server 关系型数据库进行开发。本次汇报我们将从系统架构、权限安全、高并发防超卖处理以及特色信用惩罚机制等维度，向各位老师展示我们后端的业务逻辑与技术闭环。下面由我们团队开始汇报。”

### Slide 2: 项目背景与系统概述
*   **建议发言**：
    > “首先是项目背景。现代酒店预订系统不仅要提供基本的展示和下单功能，还要应对各种异常业务场景。Luxe Stay 定位于中高端预订，重点攻克了三个业务痛点：一是高并发抢房时的库存扣减一致性；二是用户频繁取消占房导致的恶意退订行为；三是敏感管理页面的非授权越权访问。针对这些痛点，我们设计了多天连续预订库存校验算法和动态信用惩罚逻辑。”

### Slide 3: 系统架构与技术选型
*   **建议发言**：
    > “在技术架构上，系统采用了经典的 MVC 三层架构。
    > 前端使用 Thymeleaf 模板引擎和 Bootstrap 5，我们通过 Custom CSS 打造了沉浸式的高端暗黑风格；
    > 后端控制层使用 Spring Boot 3，并编写了轻量级的 `LoginInterceptor` 拦截器做安全校验；
    > 在持久层，我们结合 Spring Data JPA 与 Hibernate 的 ORM 映射，使用 `@Transactional` 保障多表数据操作的原子性；
    > 数据库选用 SQL Server，通过外键、联合唯一键和 CHECK 约束，从物理结构上确保了基础数据的绝对一致性。”

### Slide 4: 用户管理与权限拦截机制
*   **建议发言**：
    > “接下来是权限安全设计。很多系统往往只做前端按钮的条件隐藏，但这无法抵御用户手动修改浏览器地址栏的越权尝试。
    > 为此，我们在后端注册了 `LoginInterceptor` 拦截器。它不仅能检验用户是否登录（未登录自动重定向到登录页），而且会对以 `/admin` 开头的所有后台路由进行垂直权限校验。非管理员角色一旦强行访问，拦截器会直接截断请求，引导其重定向到 `403` 无权限页面，实现了严密的垂直安全隔离。”

### Slide 5: 多天预订与跨天库存校验
*   **建议发言**：
    > “我们的核心业务支持多天预订。当用户预订 3 天或更多天时，系统不是简单地只看第一天。
    > 我们的算法会根据用户选择的时间区间，逐日循环查询并校验 `room_inventory` 每日库存表。如果其中任意一天的剩余库存不足以支持预订，整个校验失败。
    > 在更新库存时，我们采用批量持久化，并由 Spring 的 `@Transactional` 进行事务保护。如果中途因高并发或库存耗尽导致扣减失败，整个预订流程全部自动回滚（Rollback），确保不会产生部分天数预订成功、部分天数预订失败的脏数据。”

### Slide 6: 信用惩罚与违约禁订逻辑
*   **建议发言**：
    > “为了防御恶意占房退订，我们首创了信用防护与自动禁订恢复机制。
    > 用户如果手动取消订单，会增加其累计取消计数器。一旦该次数累加达到 3 次，系统会将其 `bannedUntil` 属性设定为当前时间加上 7 天的自动封禁惩罚。在此期间，用户无法发起任何新订单。
    > 此外，当 7 天的禁订期满后，在用户下次尝试下单时，系统底层检测到时效已过，会自动清空其 `cancelCount` 并解除禁订状态，实现系统全自动的状态扭转。同时我们规定必须提前一天取消订单，入住当天不可取消，进一步保障了商家的利益。”

### Slide 7: 高并发安全与越权防范设计
*   **建议发言**：
    > “在高并发安全方面，传统的代码层面判断在瞬时高流量下会失效。我们通过两道防线解决超卖：一是采用 JPA 的行级排他锁（Pessimistic Write）机制，在读取某天库存时阻塞其他并发事务；二是在 SQL Server 的库存表上设立了 `available_quantity >= 0` 的物理 CHECK 约束，双重杜绝并发导致的负数超卖。
    > 另外，针对开发后期遇到的旧数据库 CHECK 约束和新增订单状态冲突的问题，我们设计并执行了本地数据库更新迁移脚本 `migrate_check_constraint.sql`，安全地重建了约束并保留了测试数据。”

### Slide 8: 团队协作与项目总结
*   **建议发言**：
    > “最后，我们团队在开发中严格遵循了 Git 多分支规范。每位成员分别在 `feature/user` 等独立分支上书写代码，通过在 GitHub 网页端发起 Pull Request 并进行 Code Review 流程后，才将稳定代码并入 `main` 主分支。
    > 综上所述，本项目在技术实现上强调强事务隔离，在业务设计上具备完善的信用机制，极佳地支撑了课程目标的达成。以上是我们的汇报，感谢老师，请老师提问！”

## 第二部分：答辩高频 Q&A 避坑指南

### Q1：你们系统的并发超卖问题是如何从根本上防范的？如果两个用户同时抢最后一间房会发生什么？
*   **评委提问意图**：考察对高并发多线程场景下的共享数据一致性、数据库锁和事务隔离的理解。
*   **推荐解答示例**：
    > “老师，在并发抢房时，如果两名用户在同一毫秒查询库存，若只在 Java 业务层判断库存是否大于 0，会因为读取到相同的库存值而同时扣减，从而导致超卖。
    > 我们系统从两层进行根本防范：
    > 1. **数据库物理约束层**：我们在 SQL Server 表的定义中设立了 `CHECK (available_quantity >= 0)`。如果高并发写入让库存扣成负数，数据库引擎会立即阻断并报错。
    > 2. **JPA 的悲观行级锁/事务层**：在查询库存记录时，我们声明了 `@Lock(LockModeType.PESSIMISTIC_WRITE)`，转化为 SQL 语句就是 `SELECT ... WITH (UPDLOCK)`。这会直接锁定当前查询的行，另一名用户的事务将进入排队等待，在上一笔事务执行完提交后才能读取更新后的库存。配合 `@Transactional` 事务的隔离性，可以从底层杜绝超卖发生。”

### Q2：详细说一下跨天连续预订的库存扣减逻辑。如果用户要订 7 月 15 到 18 号共 3 晚，中间 16 号满房了，系统会怎么处理？
*   **评委提问意图**：考察复杂业务逻辑的代码逻辑编写能力，以及事务一致性的应用。
*   **推荐解答示例**：
    > “当用户选择 7 月 15 日至 18 日预订时，系统会计算出需要占用的日期为 15、16、17 日三晚。
    > 业务流程为：
    > 1. 在 `BookingService` 中，系统会通过循环（或者根据起止日期区间）读取 `room_inventory` 表中这 3 天对应此房型的记录。
    > 2. 当遍历到 16 号时，发现其 `available_quantity` 已经是 0，无法满足预订数量。
    > 3. 此时，代码会立即抛出 `IllegalArgumentException`，阻止后续步骤。
    > 4. 由于整个预订方法被注解了 `@Transactional`，Spring 的事务管理器捕获到这个运行时异常后，会强制使之前 15 号已被锁定或扣减的库存状态**全部自动回滚（Rollback）**，确保订单绝不会创建，数据库没有任何脏数据留存。”

### Q3：为什么不使用专业的 Security 框架（如 Spring Security / Shiro），而手写 `HandlerInterceptor` 拦截器？它有什么好处？
*   **评委提问意图**：考察选型决策，评估对于项目复杂度与技术栈适配的把控。
*   **推荐解答示例**：
    > “我们进行这个技术选型主要是基于项目复杂度和运行效率的考量：
    > 1. **轻量与易维护**：本项目作为课程设计，其权限模型是典型的基于角色的垂直隔离（普通用户与 ADMIN 管理员）。使用 Spring Security 会引入大量的过滤器链配置和安全上下文开销，对于中小型课设有些过度设计。
    > 2. **可控性强**：手写 `HandlerInterceptor` 使得我们能在 `preHandle` 方法中直接控制状态，可以精准地对未登录、未授权路由进行重定向或转发（例如跳转到 403 页面），在本地调试和排查问题时更加简单透明，非常适合我们目前的开发阶段。”

### Q4：在开发过程中，听说你们本地运行去支付操作时遇到了 CHECK 约束报错，怎么回事？你们是怎么解决的？
*   **评委提问意图**：考察解决实际数据库迁移冲突的 Debug 能力。
*   **推荐解答示例**：
    > “是的老师。在开发后期，我们将订单增加了 `'PAID'`（已支付）、`'CHECKED_IN'`（已入住）和 `'COMPLETED'`（已完成）等精细化状态。但因为我们本地数据库是较早前根据老版实体类创建出来的，当时数据库内残留的 CHECK 约束只支持旧状态（如 `BOOKED` 和 `CANCELLED`）。
    > 即使在 Spring Boot 中配置了 `ddl-auto: update`，但 **Hibernate 自动更新只能增删表或列，无法去重写数据库中已有的 CHECK 约束**，这导致更新状态为 `PAID` 时，触发了 `DataIntegrityViolationException`。
    > 为了解决这个冲突，我们设计并执行了本地迁移脚本 `migrate_check_constraint.sql`。在脚本中，我们通过系统表 sys.check_constraints 查询到旧约束的名称，使用 DDL 语句 `DROP CONSTRAINT` 强行删除了它，并创建了包含全部 5 种状态在内的新约束 `CK_booking_order_status`。这样在不丢失已有本地测试数据的情况下，安全地完成了本地数据库升级，解决了报错问题。”

### Q5：如何防止用户通过直接在 URL 输入其他人的订单 ID，来偷看或操作别人的订单（即水平越权攻击）？
*   **评委提问意图**：考察对于应用开发中常见的安全风险（水平越权）的防护意识。
*   **推荐解答示例**：
    > “为了应对水平越权，我们系统采取了以下两个策略：
    > 1. **强制关联用户上下文**：在查询或操作订单的 API（如支付订单 `/orders/{id}/pay`）的 Controller 方法中，我们不会只拿着前端传来的 `orderId` 去数据库查询。我们会在业务层同时传入当前登录用户的 `userId`（通过 Session 获取）。
    > 2. **底层 SQL 限制范围**：在 `bookingOrderRepository` 中，我们使用的是根据 `id` 和 `userId` 共同查询的方法：`findByIdAndUserId(orderId, userId)`。这样就算恶意用户手动篡改了 URL 里的订单 ID，因为该订单的 `user_id` 与他当前登录的账号不匹配，数据库依然会返回空数据并抛出『订单不存在或无权操作』的异常，从根本上防止了水平越权。”
