01 / 项目背景 Luxe Stay

项目背景与系统概述

针对现代酒店预订中的高并发抢房、恶意取消订单及权限访问漏洞，提供一站式业务解决方案。

系统业务定位

Luxe Stay 定位于中高端雅奢酒店预订。除了提供常规的房型展示、库存管理、在线下单流程外，系统围绕核心教学与业务指标，重点落实了以下业务规则：

  • 多天连续预订：系统支持用户一次性选择预订多天，并在下单时对选择区间内的每日库存执行严格校验。
  • 信用惩罚防护：针对频繁恶意取消订单的行为，提供动态锁定的信用限制规则。
针对性的三大业务痛点
  • 高并发下的超卖问题：防止同一间房在同一时间被多名用户抢单成功，造成库存倒扣。
  • 恶意频繁退订风险：避免预订库存被长期占用后退订，影响酒店的实际入住率与收益。
  • 后台管理越权风险：防范普通用户通过手动篡改 URL 绕过前端导航，非法侵入后台管理系统。
02 / 系统架构 Luxe Stay

系统架构与技术选型

系统基于经典的 MVC 分层架构，选用成熟且高效的技术栈，确保高可用与高一致性。

前端渲染层

Thymeleaf & Bootstrap 5

  • 动态模板引擎同步渲染，避免单页应用的首屏加载延迟。
  • 自适应响应式栅格，适配手机与 PC 浏览器。
  • 引入 Custom CSS，打造暗黑高端玻璃拟态视觉风格。
业务与控制层

Spring Boot 3.x

  • 基于 MVC 分层架构，统一控制器逻辑。
  • 自定义 `LoginInterceptor` 拦截器，自动校验 Session 状态。
  • 提供全局异常处理机制，保障线上服务的优雅降级。
数据持久层

Spring Data JPA

  • 基于 Hibernate 6 进行高效的对象关系映射（ORM）。
  • 使用 `@Transactional` 保障多步写操作的事务原子性。
  • 利用 Repository 接口，支持底层锁机制和乐观锁版本标记。
数据存储层

SQL Server 2025

  • 使用 Microsoft SQL Server Express 作为关系型数据库。
  • 利用外键约束与联合唯一键（`UQ_room_inventory`）防止脏数据。
  • 启用强 CHECK 约束，保证状态字段的字段一致性。
03 / 权限安全 Luxe Stay

用户管理与权限拦截机制

利用拦截器在后端构建第一道安全防火墙，实现普通用户与管理员的严格角色隔离。

拦截器 (Interceptor) 工作流

系统未采用复杂的安全框架，而是使用轻量级的高效的 **`HandlerInterceptor`** 进行安全把关：

  • 注册拦截器：在 `WebConfig` 中配置拦截器拦截除登录、注册、首页及静态资源外的所有请求。
  • 双重权限隔离：
    • 第一层：校验 Session 中是否存在登录用户（未登录重定向至 `/login`）。
    • 第二层：对于 `/admin/**` 路由，额外校验 `User.getRole()` 是否为 `'ADMIN'`，若非管理员则抛出权限异常重定向至 `403.html`。
@Component
public class LoginInterceptor implements HandlerInterceptor {
    @Override
    public boolean preHandle(HttpServletRequest req, HttpServletResponse resp, Object h) {
        User user = (User) req.getSession().getAttribute("user");
        if (user == null) {
            resp.sendRedirect("/login");
            return false;
        }
        // 后端拦截管理员路由，防范越权
        if (req.getRequestURI().startsWith("/admin") && !"ADMIN".equals(user.getRole())) {
            resp.sendRedirect("/403");
            return false;
        }
        return true;
    }
}
04 / 核心业务 Luxe Stay

多天预订与跨天库存校验

支持跨天预订，使用严格的逐日校验算法，在事务控制下保障库存扣减的准确性。

1

查询并锁定范围

用户选择入住与离店日期，限制预订范围在未来 7 天内。

2

逐日库存校验

循环获取区间内每一天的 `RoomInventory` 记录，校验剩余数量是否充足。

3

扣减库存并快照

扣减对应日期的库存，创建订单并保存下单时的单价快照。

4

事务原子提交

`@Transactional` 保护。期间任何一天库存不足，全流程自动 Rollback。

跨天扣减机制的优势

传统的单日扣库存无法应对跨天场景。Luxe Stay 通过在 `room_inventory` 表中设计 `(room_type_id, inventory_date)` 的联合唯一索引，并使用 `saveAll()` 批量持久化更新，将多天校验时间复杂度控制在 O(N)。一旦中途库存扣减冲突，数据库的约束和 Spring 事务会立刻撤销该笔订单，彻底杜绝数据不一致。

05 / 信用机制 Luxe Stay

信用惩罚与违约禁订逻辑

首创“取消限额 + 自动禁订期限”的信用防护机制，最大限度保障商家免受恶意占房带来的财务损失。

3 次
累计取消上限
当用户手动取消已预订的订单时，系统将通过 `userService` 累加该用户的 `cancelCount`（取消订单数）。
7 天
预订禁止惩罚期
当取消次数达到 3 次时，系统会自动将用户的 `bannedUntil` 属性设定为当前时间往后顺延 7 天。
自动
惩罚解除与清零
当禁订期满后，用户再次尝试下单时，系统会自动清零该账号的取消次数，并自动恢复正常的预订权限。
取消时间限制约束

除了取消次数外，系统规定用户至少需要提前 1 天取消订单。如果当前系统时间已到达入住当天，则用户端的“取消订单”按钮将不再允许发起操作。这有效防止了用户在临近入住前恶意占用房间库存的情形。

06 / 系统安全 Luxe Stay

高并发安全与越权防范设计

在底层构建多重锁保护，保障高并发请求下共享资源的读写安全，并强化越权防范。

高并发抢房安全保障

在高并发场景下（如两名用户同时提交最后一个房间），单纯依靠 Java 代码中的 if判断极易发生“超卖”：

  • 行级锁保护 (Pessimistic Write)：在底层 JPA 查询每日库存记录时，采用行锁机制，阻塞其它并发事务对同房型同日期库存的修改。
  • 数据库强约束拦截：在 `room_inventory` 表上设有 `available_quantity >= 0` 的 CHECK 约束。若并发导致库存扣减至负数，数据库将直接抛出异常，引发 Spring 事务回滚。
越权防范与本地约束迁移
  • 后端路由强制拦截：除了前端对“管理后台”按钮的条件隐藏外，所有 `/admin/**` 请求必须经由拦截器做二次角色比对。任何企图手动通过 URL 越权的行为都会被抛回 403 页面。
  • 本地数据库约束迁移： 针对本地开发环境下旧的数据库约束（如 `CK__booking_o__status`）引发的支付报错，通过运行专门的迁移脚本，重构了包含全部 5 种状态在内的完整字段约束，保障数据一致性。
07 / 团队总结 Luxe Stay

团队协作与项目总结

本项目是一个符合现代企业开发标准的多分支协作和严谨业务闭环的工程实践。

Git 多分支规范协作

拒绝在 `main` 分支上直接修改代码。团队成员基于各自功能点拉取 `feature/xxx` 分支在本地草稿纸上开发，开发完毕推送后，统一在 GitHub 上发起 Pull Request，评审无误后合并。这确保了主干代码库的稳定与清洁。

严谨的业务闭环

从用户注册、多天库存严格扣减，到防恶意退订的信用自动惩罚机制，再到管理员后台的安全越权防御，项目在后端形成了完整的、自洽的业务逻辑闭环，所有状态流转均受到底层数据库与应用级事务的严密防护。

课设答辩核心支撑

本项目通过并发安全演练（抢房超卖控制）、时间窗口取消限制、以及信用期满自动恢复等场景的真实测试，全面回应了指导老师在重点和难点部分的提问，极佳地支撑了课程目标 1 和 2 的达成。
