# 酒店在线预订平台开发起步与 Git 协作指南

本指南旨在帮助开发团队明确项目开发方向、理清首日准备工作，并重点为初学者提供详细的 **GitHub 多分支协作与代码提交指南**，以确保团队开发的高效与代码合并的顺畅。

---

## 一、 项目成品功能与课设要求对照

根据课程设计的要求，本项目最终实现的成品将是一个**具备完整信用惩罚机制的酒店在线预订平台**。

### 1. 核心功能模块清单
*   **用户管理模块**：
    *   **注册与登录**：支持普通用户与管理员的账号注册、登录及登出。
    *   **权限拦截**：对普通用户和管理员进行角色隔离，防止普通用户通过直接输入 URL 访问管理员后台。
*   **房型管理模块（管理员端）**：
    *   **房型 CRUD**：管理员可以对房型信息进行增加、删除、修改和查询（包含名称、描述、价格、图片路径等）。
    *   **每日库存配置**：管理员可设定并管理具体日期（如 7 月 13 日）某房型的可用数量。
*   **房型查询与展示（用户端）**：
    *   **按日期范围筛选**：用户可选择入住与离店日期（限制在未来一周内），系统实时筛选在该日期范围内每天都有库存的房型。
    *   **详情展示**：展示房型的精美图片、单价、库存剩余状态等。
*   **预订与订单模块**：
    *   **多天预订扣库存**：支持跨天预订，系统自动校验入住日至离店日前每一天的库存。
    *   **订单快照**：订单生成时记录下单时的单价，防止后续管理员调价影响历史订单。
    *   **事务保护**：预订与取消均使用数据库事务（`@Transactional`），确保订单生成与库存扣减同时成功或失败。
*   **取消预订模块**：
    *   **时间限制校验**：用户必须提前至少一天取消订单（例如：当前日期必须早于入住日期，入住当天不可取消）。
    *   **库存释放**：取消成功后，对应日期范围内的房型库存自动加回。
*   **信用管理模块**：
    *   **取消次数累计**：记录用户的订单取消次数。
    *   **一周禁订限制**：累计取消达 3 次后，自第 3 次取消时间起， 7 天内禁止该用户发起任何新预订；禁订期满后，取消计数器清零并恢复预订权限。

## 二、 项目第一天启动指南

第一天的核心目标是：**统一标准、建好仓库、定好数据库，并让每个人都能成功运行起基础项目。**

### 1. 第一天工作流时间线
```text
【上午：对齐规范】 ───> 【中午至下午：搭建骨架】 ───> 【下午至傍晚：全员跑通】 ───> 【晚间：分工与准备】
  - 确定技术版本         - 组长拉取 Spring Boot 项目    - 队员 Clone 代码             - 分配具体接口
  - 规范命名与 Git 规则   - 编写初始 SQL 建表与测试数据   - 导入本地 SQL 跑通服务       - 引入前端模板
```

### 2. 第一天具体任务清单
1.  **环境对齐**：全员统一 JDK 版本（如 JDK 17）、SQL Server 版本、Maven 版本和 IDE。
2.  **创建数据库脚本 (`schema.sql` & `data.sql`)**：
    *   由一人写好建表脚本（包括 `users`、`room_type`、`room_inventory`、`booking_order` 及其主外键/联合唯一约束）。
    *   准备测试数据：创建 1 个管理员账号、1 个用户账号、3 个测试房型及未来 7 天的每日库存数据。
3.  **新建 Spring Boot 工程**：
    *   依赖项引入：`Spring Web`、`Thymeleaf`、`Lombok`、`Spring Data JPA`、`MS SQL Server Driver`。
    *   编写好 `application.yml` 的数据库连接配置。
4.  **初始化前端静态资源**：
    *   在 `src/main/resources/static/css` 下放置公共 CSS，引入 Bootstrap 框架。
    *   在 `templates/fragments/` 下编写通用的导航栏 `header.html` 和页脚 `footer.html`。

## 三、 GitHub 多分支协作超详细教程

对于没有 Git 协作经验的同学，多分支开发是保证“代码不打架”的关键。

### 1. 核心概念：“草稿纸”与“合订本”
*   **`main` 分支（主分支/合订本）**：代表“正式发布、稳定运行的代码”。不能直接在上面修改。
*   **`feature/xxx` 分支（个人分支/草稿纸）**：每个人在自己的分支上写代码。例如：
    *   组员 A 在 `feature/user` 开发用户登录。
    *   组员 B 在 `feature/room` 开发房型管理。
    *   组员 C 在 `feature/booking` 开发订单系统。
*   **合并（Merge）**：当你在自己的分支开发完毕并测试通过后，再申请把你的代码合并到 `main` 分支。

---

### 2. 团队协作常用 Git 指令速查（命令行版）
如果你喜欢使用终端/命令行，可以参考以下基本操作流程：

#### ① 初始化与克隆项目
```bash
# 从 GitHub 克隆项目到本地
git clone <你的项目GitHub仓库地址>
cd <项目文件夹>
```

#### ② 创建并切换到自己的新分支
```bash
# 每次写新功能前，先切换到 main 并更新到最新
git checkout main
git pull origin main

# 基于最新的 main 分支，创建并切换到你自己的分支（如 feature/user）
git checkout  feature/room
git checkout -b feature/room

```

#### ③ 编写代码并提交到本地仓库
```bash
# 查看有哪些文件被修改了
git status

# 将修改过的文件加入暂存区（. 代表所有文件）
git add .

# 提交到本地仓库，并附带说明信息
git commit -m "完成了模块b"
```

#### ④ 推送分支到 GitHub 远程仓库
```bash
# 将你的本地分支推送到 GitHub（第一次推送时需要 -u，后续直接 git push）
git push -u origin feature/room
```

### 3. IntelliJ IDEA 界面化操作指南（推荐初学者使用）

IDEA 提供了极其直观的图形化 Git 界面，绝大多数操作都可以通过鼠标点击完成。

#### 第一步：新建你自己的开发分支
1. 在 IDEA 右下角，找到写着 **`Git: main`** 的状态栏并点击它。
2. 在弹出的菜单中，选择 **+ New Branch**（新建分支）。
3. 命名为 `feature/user`（以你的模块为准）。
4. 确保勾选了 **Checkout branch**（切换分支），然后点击 **Create**。
5. *此时右下角的状态显示为 `Git: feature/user`，代表你已安全进入自己的“草稿纸”空间。*

#### 第二步：提交代码 (Commit)
当你写完了一个小功能（例如写好了 `UserController.java`）：
1. 使用快捷键 `Ctrl + K`（或点击右上角的绿勾图标，或点击菜单栏 **Git -> Commit...**）。
2. 在弹出的 **Commit** 窗口中，勾选你想要提交的文件。
3. 在 **Commit Message** 输入框中，写下这段代码做了什么，例如：`feat: 增加用户登录密码 BCrypt 加密处理`。
4. 点击下方的 **Commit** 按钮。
5. *注意：此时代码仅仅保存在你的本地硬盘中，GitHub 上依然没有。*

#### 第三步：推送至 GitHub 仓库 (Push)
1. 使用快捷键 `Ctrl + Shift + K`（或点击右上角斜向上的绿色箭头，或点击菜单栏 **Git -> Push...**）。
2. 在弹出的 Push 窗口中，确认左侧显示的是 `feature/user` 分支。
3. 点击 **Push** 按钮。
4. *等待进度条完成，此时你的代码已经成功备份到了 GitHub 远程服务器。*

#### 第四步：发起合并请求并合并 (Pull Request / PR)
当你完成了该模块的所有开发，需要将代码合并到 `main` 主分支：
1. 打开项目的 **GitHub 网页端**。
2. 登录后进入仓库首页，网页上方会出现一个黄色的提示条：`feature/user had recent pushes less than a minute ago`。
3. 点击黄条右侧的绿色按钮 **Compare & pull request**。
4. 填写本次合并的标题和详细说明，然后点击 **Create pull request**。
5. **代码评审**：邀请团队其他成员打开这个 Pull Request 页面，查看你的代码修改。
6. **合并代码**：确认无误后，点击绿色的 **Merge pull request** 按钮，再点击 **Confirm merge**。
7. *至此，你的代码正式并入主分支 `main`！*

### 4. 每日开发同步工作流（避免冲突的黄金法则）

为防止多人开发同一文件产生“代码冲突”，请全员务必严格遵守**“每日三步曲”**：

#### 早上开始工作前（拉取最新代码）：
1. 在 IDEA 右下角，将分支切换回 **`main`**。
2. 点击 **Pull** 蓝色下箭头，获取队友昨天并入的最新代码。
3. 切换回你自己的分支 **`feature/xxx`**。
4. 在右下角点击 **`main`** 分支，在弹出菜单中选择 **Merge 'main' into 'feature/xxx'**（将最新的 main 合并到我的分支）。
5. *这能确保你在最新的代码基础上开始新一天的开发。*

#### 下午开发过程中（勤提交）：
* 每写完一个小的、可以运行的步骤，就执行一次 **Commit**。
* 这样即便后面写挂了，也可以随时右键 Rollback 回退到上一个健康的 Commit。

#### 晚上准备合并前（本地测试）：
1. 申请合并前，自己先在本地运行一下项目，确保没有报错，功能完全正常。
2. 在 GitHub 发起 PR，合并到 `main` 后，记得通知群里其他成员：“我已经合并了订单功能，请大家更新一下 `main` 分支”。

## 五、 重点测试场景与答辩演示准备

在项目开发接近尾声时，团队应合力完成以下关键场景的测试，这也是答辩中评委老师最常提问的地方：

1.  **跨天库存校验**：测试预订 7.13 ~ 7.15（共两晚），如果 7.13 有房，但 7.14 满房，系统能否正确提示“库存不足”并阻止预订？
2.  **并发抢房测试**：两名用户同时提交同一个房型的最后一间空房，系统是否会发生超卖（即一间房被两个人订走）？*（解决方案：使用 `@Transactional` 及数据库乐观锁或行级锁保护）*。
3.  **违约禁订逻辑**：手动让一个测试用户取消 3 次订单，观察该账号在接下来的 7 天内再次提交订单时，系统是否能正确弹出“已被禁止预订”的提示。
4.  **越权防范测试**：使用普通用户账号登录后，在浏览器地址栏手动输入 `/admin/room/list`，系统是否能正确重定向到 `403 Forbidden` 或登录页？
