# 酒店在线预订平台开发建议

## 一、项目技术方案

推荐采用：

> Spring Boot + Thymeleaf + Bootstrap + JavaScript + SQL Server

各项技术的作用：

- **Spring Boot**：实现用户、房型、库存、订单和权限等后端业务。
- **Thymeleaf**：把后端数据直接显示在 HTML 页面中，不需要使用 Vue。
- **HTML/CSS**：负责页面结构和样式。
- **Bootstrap**：快速制作导航栏、表格、表单、按钮和响应式页面。
- **JavaScript**：实现日期选择、表单校验和取消确认等简单交互。
- **SQL Server**：保存用户、房型、每日库存和订单数据。

建议开发一个 Spring Boot 单体网站，不采用前后端分离，以降低学习和联调难度。

## 二、核心业务流程

第一阶段应优先完成：

```text
注册/登录
    ↓
查询可用房型
    ↓
提交预订
    ↓
校验并扣减库存
    ↓
查看订单
    ↓
取消订单
    ↓
恢复库存
```

核心流程稳定后，再增加：

- 管理员房型管理
- 房型图片上传
- 取消次数统计
- 三次取消后一周内禁止预订
- 页面美化和移动端适配

暂时不要加入支付、短信、地图和评价等非必要功能。

## 四、数据库设计建议

### 1. 用户表 `users`

| 字段 | 说明 |
| --- | --- |
| `id` | 用户主键 |
| `username` | 登录账号 |
| `password` | 加密后的密码 |
| `real_name` | 真实姓名 |
| `phone` | 手机号 |
| `role` | `USER` 或 `ADMIN` |
| `cancel_count` | 取消次数 |
| `banned_until` | 禁止预订截止时间 |
| `created_at` | 注册时间 |

注意：

- 不要将表命名为 `user`，避免与数据库关键字冲突。
- 密码不能明文保存，建议使用 BCrypt 加密。

### 2. 房型表 `room_type`

| 字段 | 说明 |
| --- | --- |
| `id` | 房型主键 |
| `name` | 房型名称 |
| `description` | 房型介绍 |
| `price` | 每晚价格 |
| `total_quantity` | 总房间数量 |
| `image_path` | 图片路径 |
| `status` | 是否启用 |

房型示例：标准间、大床房、双床房、豪华套房。

### 3. 每日库存表 `room_inventory`

| 字段 | 说明 |
| --- | --- |
| `id` | 库存记录主键 |
| `room_type_id` | 对应房型 |
| `inventory_date` | 库存日期 |
| `available_quantity` | 当天剩余数量 |

建议为房型和日期建立联合唯一约束：

```sql
UNIQUE (room_type_id, inventory_date)
```

这样能保证同一房型在同一天只有一条库存记录。

### 4. 订单表 `booking_order`

| 字段 | 说明 |
| --- | --- |
| `id` | 订单主键 |
| `order_no` | 订单编号 |
| `user_id` | 下单用户 |
| `room_type_id` | 预订房型 |
| `check_in_date` | 入住日期 |
| `check_out_date` | 离店日期 |
| `quantity` | 预订数量 |
| `unit_price` | 下单时单价 |
| `total_amount` | 订单总金额 |
| `status` | 订单状态 |
| `created_at` | 创建时间 |
| `cancelled_at` | 取消时间 |

订单状态建议使用：

- `BOOKED`：已预订
- `CANCELLED`：已取消
- `COMPLETED`：已完成

订单必须保存下单时的单价。管理员以后修改房价时，不能影响历史订单。

## 五、关键业务规则

### 1. 离店日期不占用库存

例如用户在 7 月 13 日入住、7 月 15 日离店，实际占用：

- 7 月 13 日库存
- 7 月 14 日库存

不占用 7 月 15 日库存。

### 2. 多天预订必须逐日检查

用户预订多晚时，系统必须检查入住日至离店日前的每一天。

只有每天的库存都充足，才能统一扣减库存并创建订单。

### 3. 预订必须使用事务

创建订单和扣减库存属于一次完整操作：

```java
@Transactional
public void createBooking(...) {
    // 检查用户是否被禁订
    // 校验入住和离店日期
    // 校验每天的库存
    // 扣减每天的库存
    // 创建订单
}
```

任何一步失败，整个操作都应该回滚，避免出现订单和库存不一致。

取消订单也需要事务，以下操作必须同时成功：

- 更新订单状态
- 恢复每日库存
- 更新用户取消次数

### 4. 防止重复取消

只有 `BOOKED` 状态的订单可以取消。

已经取消的订单不能再次取消，否则会重复恢复库存。

### 5. 取消时间限制

建议规定：

> 当前日期必须早于入住日期，入住当天不能取消。

题目中要求“至少提前一天取消”，具体计算规则最好提前与老师确认。

### 6. 信用管理规则

题目中的“连续三次取消”存在一定歧义，建议和老师确认。

为了降低实现难度，可以定义为：

> 用户累计取消三次后，从第三次取消时间开始，七天内禁止创建新预订；禁订结束后取消次数清零。

使用以下字段实现：

```text
cancel_count
banned_until
```

## 七、项目结构建议

```text
src/main/java/项目包名/
├─ controller
├─ service
├─ repository
├─ entity
├─ dto
├─ config
└─ exception

src/main/resources/
├─ templates
│  ├─ user
│  ├─ room
│  ├─ booking
│  ├─ admin
│  └─ fragments
├─ static
│  ├─ css
│  ├─ js
│  └─ images
└─ application.yml
```

代码调用关系保持为：

```text
Controller → Service → Repository → SQL Server
```

Controller 负责接收请求和返回页面，复杂业务逻辑应写在 Service 中。

## 八、推荐开发顺序

### 第二阶段：并行开发

- 组员 A 完成注册、登录和权限控制

### 第三阶段：核心流程联调

- 用户登录后查询房型
- 选择日期并提交预订

## 九、Git 协作建议

建议使用以下分支：

```text
main
feature/user
feature/room
feature/booking
```

协作规则：

- 每个人在自己的功能分支开发。
- 每完成一个小功能就提交一次。
- 合并前先运行和测试自己的模块。
- 不要三个人同时修改同一个公共文件。
- `application.yml`、公共导航栏、全局 CSS 和数据库脚本由指定成员统一维护。
- 不要等到项目最后一天才第一次合并代码。

## 十、重点测试场景

至少测试以下情况：

1. 离店日期早于或等于入住日期。
2. 查询日期超过未来一周范围。
3. 多日预订中某一天库存不足。
4. 两名用户同时预订同一个房型。
5. 用户重复点击提交按钮。
6. 已取消的订单再次取消。
7. 入住当天尝试取消订单。
8. 取消订单后库存是否正确恢复。
9. 普通用户直接访问管理员页面。
10. 用户取消三次后是否被禁止预订。

## 十一、答辩演示建议

答辩时重点演示完整业务链：

```text
管理员新增房型并设置库存
    ↓
普通用户按日期查询房型
    ↓
用户提交预订
    ↓
系统生成订单并减少库存
    ↓
用户取消订单
    ↓
系统恢复库存并记录取消次数
    ↓
用户三次取消后触发一周禁订
```

完整、稳定的业务流程，比大量但不完善的附加页面更能体现项目质量。

## 十二、最终建议

项目应优先保证以下三点：

1. **订单与库存数据一致**：预订扣库存，取消恢复库存，并使用事务保护。
2. **权限隔离正确**：普通用户不能访问管理员功能。
3. **三人的代码能够顺利合并**：提前统一数据库、项目结构和页面风格。

先完成核心流程，再做页面美化和扩展功能，可以有效降低项目延期风险。
